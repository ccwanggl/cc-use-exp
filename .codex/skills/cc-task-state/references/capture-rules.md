# Capture Rules

抽取原则：

1. 优先找任务主体，而不是先找状态词
2. 一段话里出现多个主体时，拆成多条任务
3. 没有证据时，不推断“已经开工”

常见信号：

- `还在收尾`、`正在处理`、`还没做完`：通常是 `进行中`
- `还没有开始`、`没有落代码`、`先记一下`：通常是 `需求中` 或 `待开始`
- `被 fix 打断`、`中途切去修问题`：主任务通常转为 `被打断` 或保留 `进行中` 并补“中断记录”；插入事项按影响范围决定是否单列
- `等 XX 完成再做`、`先不写代码`：通常是 `待开始`
- `卡在`、`依赖`、`等接口/等确认`：通常是 `阻塞中`

拆分示例：

用户描述：

> 它目前状态还是进行中，主要在收尾 Codex CLI / Responses / 路由弹框这一整条链路。你刚才提的“请求日志增加站点和模型筛选搜索”这个新功能，还没有单独开新任务落代码。中间被连续几轮 fix 打断了，所以现在还停留在需求层，没有开始实现。

建议落盘：

- 任务 A：`Codex CLI / Responses / 路由弹框链路收尾`
  状态：`进行中`
- 任务 B：`请求日志增加站点和模型筛选搜索`
  状态：`需求中`

不要落成：

- 一条名为“请求日志筛选 + 路由弹框收尾”的混合任务
- 把任务 B 标成 `进行中`
